4. 风险与待办事项 (Risks & Todos)

· 技术风险：
  · iLink SDK 集成复杂性： 微信适配器的实现高度依赖 iLink SDK 的文档和稳定性。若 SDK 回调机制不稳定，asyncio.to_thread 转换可能存在性能或死锁风险。建议尽早进行原型验证 (Spike)。
  · sqlite-vec 环境兼容性： sqlite-vec 扩展在不同操作系统 (Linux/macOS/Windows) 和 Docker 镜像下的编译和加载可能存在兼容性问题，是部署前的主要环境风险。备用方案是回退到纯 FTS5 检索，但需评估记忆检索质量下降的影响。
  · AI 提供商服务可用性与速率限制： 依赖 OpenAI/Anthropic 等第三方 API，其稳定性、速率限制 (429) 和服务降级直接影响核心对话质量。单提供商故障时，需确保降级逻辑 (规则回复) 的用户体验可接受。
  · “压抑-爆发”情感模型调优： suppress_tendency, emotional_sensitivity 等参数的设定会极大影响用户感受。系统上线后可能需要频繁调整，应考虑将参数配置化，以便在无代码变更下进行 A/B 测试。
· 待确认问题：
  · 用户时区数据来源： 主动陪伴的早安/晚安触发依赖准确的用户时区，目前设计文档未明确时区信息是来自用户绑定配置、客户端上报还是 IP 推断。需产品经理/架构师确认初始获取与存储方案。
  · quiet_hours 静音时段定义： 文档中提出了静音时段概念，但未明确配置格式和默认值（如晚上10点到早上7点？），需确认。
  · night_metaphor 生理隐喻的具体内容与概率： 此功能的体验设计非常关键，稍有不慎会显得突兀。需确认是否有预设的隐喻词库和投放概率控制。
  · WebChat 用户认证： 当前 WebSocket 端点 /ws/{user_id} 设计为无认证，直接传递 user_id，存在安全隐患。需确认是否在开发初期引入简单的 Token 或 Session 认证机制。
  · 表情包文件来源与版权： 需要明确初始的表情包资源来源、存储路径规范以及版权合规确认。
